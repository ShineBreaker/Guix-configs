** 核心概念速览
在深入配置前，先理解几个 Guix 特有的概念：

*** 1. Channel（频道）
Guix 的软件包定义分布在多个「频道」中。默认只有 =guix= 官方频道，本配置额外添加了：

+ *[[https://gitlab.com/nonguix/nonguix][nonguix]]* - 提供 Linux内核、专有驱动等
+ *[[https://github.com/ShineBreaker/jeans][jeans]]* - 个人频道，包含自定义包
+ *[[https://codeberg.org/hako/rosenthal][rosenthal]]* - 提供很多用于强化 window manager 体验的相关组件

频道定义在 [[./source/channel.scm][source/channel.scm]]，版本锁定在 [[./source/channel.lock][source/channel.lock]]。

*版本锁定* 的相关文件 (=channel.lock=) 实际上利用了 channel 的参数中可以指定 *commit* 的特点，通过锁住 commit 来固定仓库

*** 2. System vs Home
Guix 有两层配置：

| 层级   | 管理范围                         | 对应命令    | 配置文件来源                     |
|--------+----------------------------------+-------------+----------------------------------|
| *System + Home* | 系统与用户配置在同一文件 | =guix system reconfigure=（内部调用 guix-home-service） | [[source/config.org]] |

*原则* ：能用 Home 解决就不用 System，保持系统层最小化。

*** 3. 利用 ~Org Mode~ 撰写配置
配置写在 =.org= 文件中，代码嵌入在 =src= 代码块里。这样做的好处：

- 配置和文档在一起
- 可以用 Emacs 的 =org-babel-tangle= 提取代码
- 阅读体验更好

本仓库使用 =maak= 自动处理 Org 文件的导出。

** 快速开始
*** 1. 安装系统（新机器）
#+begin_src sh
# 进入 Guix 安装环境后，克隆本仓库
git clone https://codeberg.org/BrokenShine/Guix-configs

# 修改 source/information.scm 中的用户名等基本信息
# 然后初始化系统
guix shell maak -- maak init
#+end_src

*** 2. 日常更新（已安装系统）
#+begin_src sh
# 查看所有可用命令
maak --list

# 常用命令
maak apply      # 应用配置（system + home 同时，一次 reconfigure）
maak rebuild    # apply + guix locate --update（重建文件索引）

# 更新软件包频道
maak update     # 更新 channel.lock 并签名提交

# 验证配置（不实际应用）
MAAK_DRY_RUN=1 maak apply

# 括号平衡检查（最轻量，不触发 guix）
maak check

**** 2. Dotfiles 组织方式 ([[./dotfiles/][dotfiles/]])
采用*两层结构*：

+ =enable/<dir>/= - 当前启用的配置：其中不可变部分（agents、desktop、desktop-suite、system、terminal、utilities）通过 Guix Home 的 =home-dotfiles-service-type= 管理为只读；可变部分（emacs 等）通过 [[https://www.gnu.org/software/stow/][GNU Stow]] 链接到用户目录，可独立修改调试
+ =disable/<dir>/= - 旧配置保留区，不再启用

#+begin_example
dotfiles/enable/
├── agents/.config/agents/                 # → ~/.config/agents/（上下文 + skills）
├── desktop/.config/niri/                  # → ~/.config/niri/
├── desktop/.config/waybar/                # → ~/.config/waybar/
├── terminal/.config/fish/                 # → ~/.config/fish/
├── system/.config/btop/                   # → ~/.config/btop/
├── utilities/.config/tmux/                # → ~/.config/tmux/（侧边栏、窗口管理）
├── desktop-suite/.config/darkman/         # → ~/.config/darkman/
└── emacs/.config/emacs/                   # → ~/.config/emacs/ (Stow)
#+end_example

*如何复用*：

在 =config.org= 中声明 immutable 配置（home-dotfiles-service-type）：

#+begin_src scheme
(service home-dotfiles-service-type
  (home-dotfiles-configuration
    (package `("desktop" "terminal" "system" ...))))
#+end_src

mutable 配置通过 Guix Home 内置的 Stow 集成自动链接到 =$HOME=，修改后运行 ~maak apply~ 重链。

**** 3. tmux 配置 ([[./dotfiles/enable/utilities/.config/tmux/][dotfiles/enable/utilities/.config/tmux/]])
tmux 配置采用 Guile 驱动 + Bash 胶水的混合架构，核心设计如下：

+ *侧边栏导航树*：由浮动 pane 渲染的层级树（Session → Window → Pane），取代原生 tab bar。通过前缀键 =Ctrl+x B= 切换显示
+ *无插件架构*：所有功能通过外部脚本 + tmux hooks 实现，无 TPM 依赖
+ *信号驱动刷新*：操作触发 SIGUSR1 即时重绘，30 秒心跳兜底
+ *CJK 感知渲染*：侧栏文本使用 Guile 实现的中文宽度计算
+ *工作区快照*：支持将当前 session 的窗口布局保存/恢复为 JSON

目录结构：
#+begin_example
~/.config/tmux/
├── tmux.conf                       # 主配置
└── scripts/
    ├── sidebar-toggle              # 侧边栏生命周期管理（Bash）
    ├── sidebar-render.scm          # 数据采集 + 渲染引擎（Guile）
    ├── sidebar-data.scm            # 兼容 wrapper → sidebar-render.scm data
    ├── git-collector.scm           # 兼容 wrapper → sidebar-render.scm git
    ├── tmux-helpers.scm            # 共享工具模块（缓存、锁、CJK 宽度）
    ├── workspace.scm               # 工作区快照管理
    ├── session-selector            # 会话选择器（fzf）
    ├── window-jump                 # 窗口跳转（fzf）
    └── which-key                   # 快捷键帮助弹窗
#+end_example

*如何复用* ：复制 =tmux.conf= 和 =scripts/= 目录到自己的配置中。需要 tmux 3.3a+ 和 Guile 支持。Guix 用户可直接在 home 环境中添加 =guile= 包。
*注意* ：侧边栏依赖 foot 终端的行为，在其他终端下可能需要调整 *default-terminal* 和 *terminal-overrides* 设置。

** 配置分层策略
理解何时修改哪个文件：

*** 添加新软件包
1. *仅用户需要*（游戏、开发工具、媒体播放器） → 在 =config.org= 的 home-services / packages 段添加
   + 添加到 =packages= 字段
   + 示例：游戏、开发工具、媒体播放器

2. *系统级必需*（文件系统工具、固件、limine 等） → 在 =config.org= 的 system-packages / system-services 段添加
   + 添加到 =packages= 字段
   + 示例：文件系统工具、系统监控、固件

*** 添加新服务
1. *用户服务*（如 swayidle、mako） → =config.org= 的 home-services 段
2. *系统服务*（如 nftables、docker） → =config.org= 的 system-services 段

*** 添加配置文件
:PROPERTIES:
:CUSTOM_ID: 添加配置文件
:END:
*优先放入 =dotfiles/=*，根据配置性质选择分层：

1. *不需要频繁修改* → 放入 =dotfiles/enable/<dir>/.config/...= 下的 immutable 类目录（agents、desktop、desktop-suite、system、terminal、utilities），由 =config.org= 的 home-dotfiles-service-type 管理
2. *需要独立调试* → 放入 =dotfiles/enable/<dir>/= 下的 mutable 类目录（emacs 等），由 GNU Stow 链接，修改后运行 =maak apply= 重链

#+begin_src sh
# 示例：添加 immutable 配置
mkdir -p dotfiles/enable/app-name/.config/app-name
cp ~/app-config.toml dotfiles/enable/app-name/.config/app-name/
# 然后在 config.org 中将 app-name 加入 home-dotfiles-configuration packages 列表

# 示例：添加 mutable 配置
mkdir -p dotfiles/enable/app-name/.config/app-name
cp ~/app-config.toml dotfiles/enable/app-name/.config/app-name/
# 下次 maak apply 运行时自动重链
#+end_src

*仅在需要动态生成时使用 source/files/* （如需要注入 Guix 软件包路径）。

** 常用任务
*** 修改用户名
编辑 =source/information.scm= ：

#+begin_src scheme
(define username "your-username")
#+end_src

然后重新运行 ~maak apply~ 。

*** 添加 Btrfs 持久化目录
编辑 =source/information.scm= ：

#+begin_src scheme
(define %data-dirs
  '(... "your-new-dir"))
#+end_src

然后在 ~config.org~ 的 Btrfs 配置部分添加对应的 ~bind-mount~ 。

*** 添加新的 Channel
编辑 =source/channel.scm= ，添加频道定义，然后运行：

#+begin_src sh
maak update
#+end_src

*** 自定义 =maak= 任务
编辑 ~maak.scm~ ，添加新函数：

#+begin_src scheme
(define (my-task)
  "我的自定义任务"
  (log-info "执行自定义任务")
  ($ `("echo" "Hello, Guix!")))
#+end_src

然后运行 ~maak my-task~ 。

** 故障排除
*** 配置声明与目录不一致
确保 =dotfiles/enable/<name>/= 目录存在，且与 =config.org= 中 =home-dotfiles-configuration= 声明的 ~packages~ 一致。

*** 系统启动后数据丢失
检查 =source/information.scm= 中的 ~%data-dirs~ 和 ~Btrfs 子卷配置~ 是否正确，确保 =/data= 分区已正确挂载。

*** Org 导出失败
确保安装了 Emacs 且启用了 =org-babel-tangle=：

#+begin_src sh
emacs --batch -l org --eval "(require 'ob-tangle)"
#+end_src

*** 配置语法检查
修改 ~*.org~ 配置后，可以用 ~dry-run~ 模式验证语法是否正确（不会实际应用到系统）：

#+begin_src sh
MAAK_DRY_RUN=1 maak apply   # 验证配置（不实际应用）
#+end_src

~maak apply~ 在执行前会自动运行括号平衡检查。也可以手动单独检查：

#+begin_src sh
maak check          # 检查配置括号平衡
#+end_src
