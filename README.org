* 核心概念速览

在深入配置前, 先理解几个 Guix 特有的概念:

** 1. Channel (频道)

Guix 的软件包定义分布在多个「频道」中。本配置除默认 =guix= 外还包含:

+ [[https://gitlab.com/nonguix/nonguix][nonguix]] — Linux 内核、专有驱动等
+ [[https://github.com/ShineBreaker/jeans][jeans]] — 个人频道 (自定义包)
+ [[https://codeberg.org/hako/rosenthal][rosenthal]] — Window Manager 增强组件 (limine 引导器、`computed-substitution-with-inputs` 等)

频道定义在 [[./source/channel.scm][source/channel.scm]], 版本锁定在 [[./source/channel.lock][source/channel.lock]]。 =channel.lock= 利用 channel 参数中 =commit= 字段锁住具体 commit, 从而固定仓库。

** 2. System + Home (统一配置)

*原则* : 能用 Home 解决就不用 System, 保持系统层最小化。

+ 在系统配置中利用了 ~guix-home-service-type~ 来导入用户配置, 在日常使用的时候利用 ~blue home~ 来 *临时* 部署home配置, 在确认无误后, 利用 ~blue rebuild~ 来将home配置 *固化* 到系统配置中。
+ 每次开机的时候, 系统会默认使用最新的配置世代, 其中会包括home配置, 这样即使误操作home配置导致部分软件无法正常工作时, 依旧可以依靠直接重启来恢复配置。

** 3. Org Mode 撰写配置

配置写在 =.org= 文件中, 代码嵌入在 =src= 代码块里:

配置和文档写在一起, 用 Emacs 的 =org-babel-tangle= 提取代码, 可以获得更好的阅读体验, 并且在 Emacs 中编辑时会更有条理。

本仓库使用[[https://codeberg.org/lapislazuli/blue][ =blue= ]]自动处理 Org 文件的导出, 以及封装一大批长指令, 具体的可以利用 ~blue list~ 查询。

* 仓库架构详解

** 关键入口

| 文件 / 目录              | 用途                                        | 修改频率                 |
|--------------------------+---------------------------------------------+--------------------------|
| =source/information.scm= | 用户名、Btrfs 子卷、持久化目录 =%data-dirs= | 初次使用时必改           |
| =source/config.org=      | 系统 + 用户配置 (唯一 Org 源)               | 经常                     |
| =blueprint.scm=          | blue 任务定义                               | 需要添加自定义任务时修改 |
| =source/channel.scm=     | 频道定义                                    | 添加/删除频道时          |

** 构建管线

#+begin_example
source/config.org
        │
        │  blue rebuild → Emacs org-babel-tangle
        ▼
tmp/config.scm
        │
        │  guix time-machine --channels=source/channel.lock system reconfigure
        ▼
新世代 (generation)
#+end_example

* 快速开始

** 1. 安装系统 (新机器)

#+begin_src sh
# 进入 Guix 安装环境后, 克隆本仓库
git clone https://codeberg.org/BrokenShine/Guix-configs

# 修改 source/information.scm 中的用户名等基本信息
# 然后初始化系统
guix shell blue -- blue init
#+end_src

** 2. 日常更新 (已安装系统)

完整列表运行 =blue list= 。 =blue --dry-run= 对任何命令生效, 仅做构建验证、不写入系统 (子进程统一短路)。

| =blue init=               | 把系统配置安装到 =/mnt= （新装机用）               | 部署 |

* 常用任务

** 修改用户名

编辑 =source/information.scm=:

#+begin_src scheme
(define username "your-username")
#+end_src

然后运行 =blue rebuild= 。

** 添加 Btrfs 持久化目录

编辑 =source/information.scm=:

#+begin_src scheme
(define %data-dirs
  '(... "your-new-dir"))
#+end_src

如需新增 Btrfs 子卷挂载, 同步编辑 =%btrfs-subvolumes=。然后在 =config.org= 的 Btrfs 配置部分添加对应的 bind-mount。

** 添加新的 Channel

编辑 =source/channel.scm= 添加频道定义, 然后运行:

#+begin_src sh
blue update
#+end_src

** 自定义 blue 任务

blue 命令通过 [[https://codeberg.org/lapislazuli/blue][blue]] 框架的 Scheme DSL 注册。普通 =(define (my-task) ...)= 不会被识别; 必须以 =(define-command (NAME-command arguments) ...)= 形式声明, 并登记到 =blueprint.scm= 末尾的 =(blueprint ... (commands (list ...)))= 列表中。

*** 模板

编辑 =blueprint.scm=, 在任意 =(define-command ...)= 块附近新增:

#+begin_src scheme
(define-command (my-task-command arguments)
  ((invoke "my-task")                       ; 用户用 =blue <此字符串>= 调用
   (category 'misc)                          ; 必填: help/maintenance/deployment/editing/stow/nix/guix/validation
   (synopsis "一句话描述")
   (help "详细帮助, 可多行"))
  ;; ----- 命令体: 任意 Scheme 表达式 -----
  (%run `("echo" "Hello, Guix!")))           ; 走 %run 自动遵循 blue --dry-run
#+end_src

*** 注册到入口

在 =blueprint.scm= 末尾的 =(blueprint ... (commands (list ...)))= 列表里追加新命令:

#+begin_src scheme
(blueprint
 (buildables (list %config-buildable))
 (testables (list %config-check))
 (commands
  (list list-command
        rebuild-command
        ;; ... 其他已有命令 ...
        structor-command
        my-task-command)))                   ; ← 追加在末尾
#+end_src

完成后运行 =blue my-task= (=blue --dry-run my-task= 仅构建验证、不真跑子进程)。

* 故障排除

** 系统启动后数据丢失

*注意*: =/= 分区是挂载在 =tmpfs= 上的,  *重启后会自动清空* , 所以需要利用 =Btrfs Subvols= 来持久化部分目录

检查 =source/information.scm= 中的 =%data-dirs= 与 Btrfs 子卷配置是否正确；确保 =/data= 分区已正确挂载。

** Org 导出失败

确保安装了 Emacs 且启用了 =org-babel-tangle=:

#+begin_src sh
emacs --batch -l org --eval "(require 'ob-tangle)"
#+end_src

** 修改过后报错

修改 ~*.org~ 配置后, 可用 =dry-run= 模式验证语法 (不会实际应用到系统) :

#+begin_src sh
blue --dry-run rebuild
#+end_src

=blue rebuild= 在执行前会自动跑整体括号平衡检查 (在 =prepare-config= 里, 对 tangle 出的 =config.scm= )。独立的 =blue check= 则做 *逐块* 检查——直接解析 =config.org= 每个 =#+NAME= 块 + 周边 =.scm= 文件, 能定位到具体出错的块名, 且不依赖 tangle。若只想验证构建、不写入系统, 用 =blue --dry-run rebuild= 。

+ 如果提示 =unbound variable=: 检查是否导入对应模块 (可以利用 [[https://search.guix.moe/][Toys]] 来查询)
+ 如果提示 =invalid field specifier=: 检查括号是否标错 (推荐用 =Emacs= 来进行检查)
+ 其余提示: 检查拼写是否错误
