## 传播与联动

写入卡片后，检查是否存在需要同步更新的关联内容：

| 写入操作                      | 检查并更新                                                       |
| ----------------------------- | ---------------------------------------------------------------- |
| 新卡片写入                    | `kb search` 检查是否与已有卡片/pattern 存在矛盾 → 有则标注或修正 |
| 新卡片涉及已有 pattern 的主题 | 检查 pattern 是否仍准确 → 过时则修补                             |
| 同类卡片累计 ≥3 张            | 评估是否应晋升为 pattern（见 `self-improving` skill 的晋升机制） |
| pattern 被引用但实际已不适用  | 立即修补或标注过期（无人维护的模式是负担）                       |
| 新卡片与其他卡片有隐含关联    | 用 `kb connect` 建立双向链接，补充 `[[file:...]]` 引用           |
| 旧卡片结论被新发现修正        | 用 `kb update` 在旧卡片追加勘误，不覆盖旧内容                    |

## 自包含原则

每张卡片应能独立理解，不依赖外部上下文：

- 标题包含结论（"X 导致 Y" 而非 "Y 问题排查"）
- 任务描述中说明问题背景和触发条件
- 代码片段标注所属文件/工具和版本
- 引用其他卡片时使用 `[[file:experiences/文件名][描述]]` 链接

## 时效性标注

对可能过时的声明（工具版本、API 行为、配置方法），在关键发现中标注验证日期：

```org
** 关键发现
- Emacs 29.4 中 treesit 语言包需手动安装（截至 2026-05，可能随版本变化）
- Guix 的 sudoers 配置在 system reconfigure 后生效（截至 2026-04）
```

未标注日期的声明视为「稳定事实」（语言特性、算法原理等不易过时的内容）。

## 矛盾检测

写入前用 `kb search` 检查是否与已有知识冲突：

- **明确矛盾**：新发现推翻旧结论 → 更新旧卡片或添加勘误，不新建矛盾卡片
- **场景差异**：新旧结论在不同条件下分别成立 → 在两张卡片中互相引用，注明各自适用场景
- **时间演化**：旧方案被新方案取代 → 更新旧卡片标注「已过时」，新卡片引用旧卡片说明演进关系

## 置信度标注

对外部声明或不确定结论，在正文中标注置信度：

| 标注       | 含义                   | 用法                       |
| ---------- | ---------------------- | -------------------------- |
| `(已验证)` | 实际测试或确认过       | 默认，可省略               |
| `(单源)`   | 仅单一来源，未交叉验证 | 文档/博客中的说法          |
| `(推测)`   | 基于已有知识推断       | 未直接验证但逻辑合理的结论 |

不确定时宁标 `(推测)` 也不要伪装确定。

## 双时态事实

当事实发生变化时，追加新信息而非覆盖：

- **事件时间**（when it happened）：实际发生的时间
- **认知时间**（when we learned it）：发现这个事实的时间

```org
** 关键发现
*** Guix swap 配置方式
- [2026-04-23] 仅 fstab 声明 swap 不够，需 operating-system 中 swap-devices
- [2026-05-01] Guix 1.5 后推荐使用 swap-space 记录，取代 swap-devices
```

旧结论不删除，标注 "已被替代" 并引用新结论。这使策展时能追溯认知演进。

## 写入后必做清单

每次写入卡片后，按顺序确认：

1. `kb lint` — 格式正确（无 Markdown 残留）
2. `kb search` — 无矛盾或重复（有则处理）
3. `kb connect` — 与相关卡片建立链接（至少检查一次关联）
4. 置信度 — 不确定结论已标注 `(推测)` / `(单源)`
5. 时效性 — 涉及版本/API/配置的内容标注了验证日期
6. 标题 — 包含结论，不是 "XX 排查" 而是 "X 导致 Y"
7. 自包含 — 脱离对话上下文仍可理解
