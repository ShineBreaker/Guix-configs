# 代码质量审查

主动搜索"代码柔道"机会 —— 能让整个分支、helpers、条件层消失的重构。不只是找局部清理机会，更要发现结构性简化。

## 审查优先级（从高到低）

1. **结构回归** — 是否有让整层复杂度消失的重构机会？能否删掉一整类 helpers、分支层或条件模式？
2. **简化机会** — 重复模式、可合并的 helpers、不必要的条件层、可内联的间接层
3. **意大利面条增长** — 新 ad-hoc 条件、纠缠的流程、随机复杂度增长
4. **边界/抽象问题** — 不必要的可选性、`any`/`unknown`、错层的逻辑、过度包装
5. **文件大小** — 超过 1000 行需有强理由；从 <1000 行推到 >1000 行是设计警告
6. **模块化** — 职责分离、耦合度、接口清晰度
7. **可读性** — 命名、注释、控制流清晰度

## 不可协商标准

- **偏向清理设计** — 不接受"能用但更乱"的代码
- **偏爱直接、无聊、可维护的代码** — 反对 hacky/魔法代码、cast-heavy 契约
- **逻辑应在规范层** — 特性逻辑不应泄露到共享路径
- **类型和边界整洁** — 质疑不必要的可选性

## 审批阻止器（发现任一则必须阻止通过）

- [ ] 有明显的可删除复杂度的 code judo 路径但未采用
- [ ] 文件从 <1000 行推到 >1000 行
- [ ] 现有流程变得更纠缠
- [ ] 添加了不必要的抽象/包装/cast-heavy 契约
- [ ] 重复已有 helper 或把逻辑放错层

## Diff 分组原则

按 reviewer 价值排序展示变更：

1. **核心逻辑** — 新行为、算法变化、状态转换、API 表面变化
   - 完整 diff + 上下文
   - 密集逻辑旁加简短伪代码摘要

2. **连接与集成** — 路由注册、依赖注入、配置接线
   - 精简版，关注"谁调谁"

3. **样板与机械变更** — 导入重排、重命名、生成代码
   - 仅文件名 + 统计列表，无内联 diff

## 复杂逻辑处理

- **伪代码摘要**：对冗长算法用 plain English 或伪代码展示，真实 diff 可折叠
- **具体示例追踪**：令人惊讶的行为变化，选具体输入对比新旧路径，标出分歧点
- **注意事项标注**：高风险/易遗漏/反直觉的块，配短标签和一句话解释

## 输出格式

```markdown
## 审查概览

- 审查范围
- 总体评估：approve / request-changes / needs-verification

## 结构回归（最高优先级）

- [发现]
- [建议]

## 关键问题

- [ ] [问题描述] — [建议修复]

## 简化机会

- [发现]

## 其他建议

- [warning/note 级别]

## 验收检查

- [ ] 结构问题已处理
- [ ] 关键问题已处理或已解释
- [ ] 测试覆盖受影响路径
```

## 防护

- 正确性/安全性/回归优先于纯风格评论
- 不写 changelog 式前言，写 reviewer 面向的分析
- 关注"为什么变"、文件间交互、diff 不明显的内容
