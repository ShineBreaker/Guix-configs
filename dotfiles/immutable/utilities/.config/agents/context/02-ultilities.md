## Git 提交相关规范

<critical>
你不是单独在这个仓库里工作。
可能还有其他 agent 或人工编辑者同时修改文件。
</critical>

### 你的行为约束如下：

1.  **commit 的相关规范必须遵守 `gitmassage` 中的描述**，该文件位于 `~/.config/git/gitmessage` 中。

2.  **操作必须遵循如下规范**：

- 把当前任务视为 **共享工作区协作任务** ，不是单人重构任务。
- 开始前必须读取 `git status --short`，并向用户说明当前工作区是否干净。
- 只允许对当前任务直接相关的文件做 **最小补丁**。
- 如果某个文件已经被修改，**必须** 先读取最新内容，再决定如何合并你的改动。
- 如果发现你的改动消失，先做 **差异定位** 和 **最小恢复** ，不要回滚整个仓库。
- 在完成前必须再次检查 `git status --short`，并确认提交内容仅包含本次任务。

3.  **注意以下禁止事项**：

- 禁止通过整文件重写来实现小功能。
- 禁止清理、格式化、重排与你任务无关的内容。
- 禁止提交无关文件。

### 如果出现以下情况，必须停止并汇报，而不是继续硬改：

- 同一文件中存在明显的并行改动冲突
- 你无法判断某段改动是否来自用户还是其他 agent
- 你需要使用破坏性 git 命令才能继续

## 环境配置相关规范

1.  项目重度依赖各种虚拟环境以及项目环境的创建，所以请你务必要：

- 用 **pnpm** 来替代 **npm**
- 用 **uv** 来替代 **pip**
- 在其他情况下：善于利用 `guix shell` ，通过撰写 `manifest.scm` 来完成环境的配置

## 知识库交互

<critical>
你接入了三个知识库相关 skill，必须在对应场景下主动使用，不需要用户提醒。
这些 skill 是你跨会话持续改进的核心机制，不是可选功能。
</critical>

### 可用 skill 一览

| Skill             | 触发场景                                                     |
| ----------------- | ------------------------------------------------------------ |
| `knowledge-base`  | 检索/写入经验卡片、管理模式、查看统计                        |
| `self-improving`  | 检测经验信号（纠正、踩坑、发现更优方案）、任务结束时自动总结 |
| `kb-curator`      | 用户要求策展、检查知识库健康度、夜间维护                     |

### 任务开始前：必须预检（CRITICAL）

<critical>
每次接到非平凡任务时，**必须** 先执行知识库预检。
这不是建议，是强制要求。
</critical>

执行步骤：

1.  调用 `knowledge-base` skill，加载 CLI 用法
2.  运行 `kb search "<任务相关的技术/工具/框架>"` 检索历史经验
3.  运行 `kb profile` 快速回顾用户偏好和活跃项目
4.  如返回高相关经验 → 作为上下文参考，在回复前输出提醒
5.  如返回空或低相关 → 静默继续

必须预检的场景：

-   涉及调试/排障（debug）
-   涉及配置修改（config）
-   涉及知识库中已有技术栈（`kb fields --tech` 查看）
-   用户描述的问题可能与之前遇到的类似

### 对话过程中：检测经验信号

在对话中自动检测以下信号，触发 `self-improving` skill 记录经验：

| 信号类型         | 示例                                         | 操作                       |
| ---------------- | -------------------------------------------- | -------------------------- |
| 用户纠正你       | "不对，应该是…"、"你搞错了…"                 | `mistake` 卡片             |
| 发现知识空白     | 文档过时、API 行为不符预期                   | `note` / `research` 卡片   |
| 踩坑/非显见错误  | 排查 \>2 步、环境差异                        | `debug` 卡片               |
| 发现更优方案     | 比初始实现更简、用户指出更佳路径             | `refactor` 卡片            |
| 配置陷阱         | 跨工具集成踩坑、非默认组合                   | `config` 卡片              |
| 飞升模式         | 同一问题被纠正 ≥2 次、用户说 `/ascended`     | 全面检索 → 最强方案 → 复盘 |

**注意防误触发**：用户只是描述报错但不纠正你、普通 review 无可复用经验 → 不触发。

### 任务结束后：必须总结（CRITICAL）

<critical>
任务结束的标志：用户说 "可以用了"、"一切正常工作"、"Done" 等表示完成的短语。
此时必须触发 `self-improving` skill 执行总结流程。
</critical>

执行步骤：

1.  评估本次对话是否有可记录的经验（参考上方信号表）
2.  如有 → 按 `self-improving` 写入流程执行（`kb search` 去重 → `kb add` → `kb lint` → 传播联动）
3.  如无 → 确认 "本次无可记录经验" 并停止
4.  如用户此时提出新要求 → **先完成总结，再处理新要求**

### 知识库策展

当用户要求检查知识库、维护整理知识库、说 "curate kb"、"夜间策展" 时，调用 `kb-curator` skill。

### 写入规范速查

-   卡片格式为 **Org mode**，不是 Markdown（`#+begin_src` 而非 ` ``` `）
-   写入前 `kb search` 去重，`kb fields` 复用已有标签
-   写入后 `kb lint` 校验格式
-   标题包含结论（"X 导致 Y" 而非 "Y 问题排查"）
-   不确定结论标注 `(推测)` / `(单源)`
-   时效性声明附 `(截至 YYYY-MM)`
-   绝不孤立写入——每次至少检查一次关联
-   发现用户偏好/习惯/项目变化时，用 `kb profile --add` 更新用户画像
