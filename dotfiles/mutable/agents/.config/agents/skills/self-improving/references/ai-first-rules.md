# AI-First 经验卡片规则

<critical>
知识库的核心读者是 AI Agent（包括未来的你），不是人类。
每张卡片都必须自包含，让任何 Agent 无需外部上下文即可理解和应用。
</critical>

## 规则 1：自包含上下文

每张卡片在脱离当前对话上下文后仍能独立理解：

- 标题包含结论（"X 导致 Y" 而非 "Y 问题排查"）
- 任务描述中说明问题背景和触发条件
- 代码片段标注所属文件/工具和版本
- 引用其他卡片时使用 `[[file:experiences/文件名][描述]]` 链接
- 不使用 "上文提到的"、"刚才那个" 等指代词

## 规则 2：AI 预备摘要

每张经验卡片在正文开头（任务描述章节之前或之中）用 1-2 句话概括：

- 这张卡片解决了什么问题
- 核心结论或关键教训是什么

示例：

```org
** 任务描述
Emacs 29 中 treesit 语言包在 Guix 上需手动安装，否则 treesit-auto 报错。
核心教训：Guix 的 emacs 包不捆绑 treesit 语法包，需独立安装。
```

## 规则 3：时效性标注

对可能过时的声明（工具版本、API 行为、配置方法），标注验证日期：

```org
** 关键发现
- Emacs 29.4 中 treesit 语言包需手动安装（截至 2026-05，可能随版本变化）
- Guix 的 sudoers 配置在 system reconfigure 后生效（截至 2026-04）
```

未标注日期的声明视为「稳定事实」（语言特性、算法原理等不易过时的内容）。

## 规则 4：置信度标注

对外部声明或不确定结论，在正文中标注置信度：

| 标注       | 含义                   | 用法                       |
| ---------- | ---------------------- | -------------------------- |
| `(已验证)` | 实际测试或确认过       | 默认，可省略               |
| `(单源)`   | 仅单一来源，未交叉验证 | 文档/博客中的说法          |
| `(推测)`   | 基于已有知识推断       | 未直接验证但逻辑合理的结论 |

不确定时宁标 `(推测)` 也不要伪装确定。

## 规则 5：传播联动

写入卡片后，检查是否存在需要同步更新的关联内容：

| 写入操作                      | 检查并更新                                                              |
| ----------------------------- | ----------------------------------------------------------------------- |
| 新卡片写入                    | `kb_mcp_kb_search` 检查是否与已有卡片/pattern 存在矛盾 → 有则标注或修正 |
| 新卡片涉及已有 pattern 的主题 | 检查 pattern 是否仍准确 → 过时则修补                                    |
| 同类卡片累计 ≥3 张            | 评估是否应晋升为 pattern                                                |
| pattern 被引用但实际已不适用  | 立即修补或标注过期                                                      |
| 新卡片与其他卡片有隐含关联    | 用 `kb_mcp_kb_connect_cards` 建立双向链接                               |

**绝不孤立写入**——每张卡片至少检查一次关联，确认无冲突或已处理。

## 规则 6：双时态事实

当事实发生变化时，追加新信息而非覆盖：

- **事件时间**（when it happened）：实际发生的时间
- **认知时间**（when we learned it）：发现这个事实的时间

```org
** 关键发现
*** Guix swap 配置方式
- [2026-04-23] 仅 fstab 声明 swap 不够，需 operating-system 中 swap-devices
- [2026-05-01] Guix 1.5 后推荐使用 swap-space 记录，取代 swap-devices
```

旧结论不删除，标注 "已被替代" 并引用新结论。这使策展时能追溯认知演进。

## 规则 7：模式即时修补

从 MEMORY.org feedback 节检索到偏好/规则并使用后，如果发现已过时、不完整或有误，必须立即修补——不要等待用户提醒。无人维护的反馈条目是负担，不是资产。

修补信号：

- 模式描述的解法对新版本不再适用
- 模式缺少关键的边界条件或反例
- 模式的前提条件已发生变化
- 有新的经验卡片可以补充到模式的引用中

## 反模式清单

| 反模式                     | 正确做法                                      |
| -------------------------- | --------------------------------------------- |
| 标题为 "XX 问题排查"       | 标题包含结论："X 导致 Y，需 Z"                |
| 正文只有流水账过程         | 提炼坑点和教训，聚焦 "没想到的地方"           |
| 依赖对话上下文才能理解     | 自包含，引用用链接而非隐式指代                |
| 不标注版本和日期           | 时效性声明附 `(截至 YYYY-MM)`                 |
| 代码片段无文件路径         | 标注 `file:line` 引用                         |
| 孤立卡片无任何关联         | 至少检查一次 `kb_mcp_kb_search`，有关联则链接 |
| 新卡片与旧卡片矛盾但不标注 | 互相引用，注明适用场景或标注替代关系          |
| 不确定的结论伪装为确定     | 标注 `(推测)` / `(单源)`                      |
