# 改写模式（expand / contract / align / merge）

`doc-engineering` SKILL.md §4 列出四种模式，本文件给出每种的工作流与**反模式**。

## 1. expand（扩写）

**触发信号**：用户说"扩写 X"、"X 太薄"、"X 没有覆盖新功能"。

工作流：

1. 跑 `scripts/doc-check.py <doc.md> --cli <cli>` 看机器可读的 drift
2. 跑工具 `<tool> --help` 取真实输出
3. 把 `help` 输出当作**结构骨架**，但**转录**成文档口吻，不要原文粘贴
4. 覆盖 4 个 lens：what / when / why / how
5. 加一节"已落地修复"或"变更记录"（这次会话的新做法）

**反模式**：

- ❌ 把整段 `--help` 输出当作文档一节粘贴
- ❌ 只扩不校——读者看到旧 + 新拼起来的内容更困惑
- ❌ 用第一人称叙事（"我们决定..."）——参考文档用陈述式第三人称

## 2. contract（缩）

**触发信号**：用户说"太啰嗦"、"重点突出"、"去掉细节"、"我说 X 没说 Y"。

工作流：

1. 先用 `wc -l` 量长度，定目标（典型 30-50% 缩）
2. 删"as of <日期>"和"by <agent name>"这种元数据——只留"as of"或"修订记录"标记
3. 删示例（保留最少 1 个 worked example）
4. 删"为什么我们这么做"叙事——保留"做什么"
5. 把表格砍到 ≤ 2 列，合并类似章节

**反模式**：

- ❌ 缩了再扩——总长不变但失去结构
- ❌ 缩了不改修订记录——读者不知道是新版本还是缩过的旧版本

## 3. align（对齐真实行为）

**触发信号**：用户说"X 文档和实际不符"、"X 工具改了但文档没改"。

工作流：

1. `scripts/doc-check.py` 跑一遍——这是机器信号
2. **逐行**对照"文档说的"vs"代码做的"——这是人工信号
3. 用 `patch()` 局部替换，不用 `write_file()` 重写（保留其他部分）
4. 在修订记录加一行"对齐 <date>: <字段> 由 X → Y"

**反模式**：

- ❌ 重写整篇来对齐 1 行——破坏其他章节
- ❌ 改了不更新修订记录——下次校对找不回这次改动

## 4. merge（合并两份文档）

**触发信号**：用户说"把 X 和 Y 合成 Z"、"X 是 Y 的子集"。

工作流：

1. 先 **读 X 和 Y 完整内容**（不读就会保留错的删对的）
2. 列出 X 与 Y 的事实交集（grep `cat X.md Y.md | sort | uniq -c` 看重复行）
3. 选 X 作为主（或 Y，由用户决定）—— 拼合
4. 用 `diff X.md Z.md` 看丢了什么；`diff Y.md Z.md` 看丢了什么
5. 把合并说明写进 Z 的"修订记录"——"由 X+Y 在 <date> 合并"

**反模式**：

- ❌ 没读两份就直接合并——保留错的删对的
- ❌ 合并后没记录"丢了什么"——下次没人能找回

## 5. 重写后的强制动作

任何模式结束都必须：

1. 跑 `scripts/doc-check.py <新 doc> --cli <cli>` 验
2. 在修订记录加一行（哪怕只写"v2.0 全文重写"）
3. 若改了 SKILL.md / templates/ / references/，re-run 该 skill 自身的 ad-hoc verify
   （skill-authoring §8.1："Verify script edited after deletion = stale"）
