# PLAN → 稳定参考文档：扩展模板与分类

`doc-engineering` SKILL.md §5 给出 3 步流程（定位 / 分类 / 改写），本文件
细化"分类"环节，并给出每类 PLAN 的具体改写规则。

## 1. 源 PLAN 四种类型与对应改法

### 1.1 缺陷清单 / 任务书（最常见）

源文件示例：`agenote-defects-2026-07-07.md`（独立任务书，D1-D5）。

**结构特征**：

- "## 环境信息" 节
- "## 缺陷清单"（D1/D2/D3...，每条带现象/根因/修复建议）
- "## 修复优先级建议"
- "## 提交清单"
- "## 修复记录（由后续 agent 执行）"

**改写目标**：在原文档（README / ref doc）加 "已落地修复" 表 + 验证摘要，
PLAN 退到 footnote 引用。

**改写规则**：

- 删"## 环境信息"——读者已经在目标文档里
- 删"## 修复优先级建议"——已落地了，无意义
- 删"## 提交清单"——用 `git log --oneline` 替代
- 删"## 修复记录" agent 节——保留"修复结果"小节作为引用
- 把每条缺陷浓缩为 "ID / 主题 / CLI 行为 / MCP 行为 / status" 一行表格

### 1.2 决策日志

**结构特征**：

- "## 选项 A / B / C" 节
- "## 选了什么 / 为什么"
- "## 拒绝的选项和理由"

**改写目标**：参考文档加 "## 决策" 节，每个决策保留一句话理由。

**改写规则**：

- 选项描述保留 1-2 句关键差异，删过程叙事
- "拒绝的选项和理由" 合并到 "考虑过的替代方案" 一行内
- 选了什么 + 理由 = 一行表格

### 1.3 工作流步骤

**结构特征**：

- "## Step 1 / 2 / 3..."
- 每步带命令或代码块

**改写目标**：参考文档加 "## 操作" 节，按编号步骤列出。

**改写规则**：

- 步骤编号必须连续
- 每个步骤命令块带"做了什么 + 期望输出"（删冗长的"我们跑了 X 然后看 Y..."）
- 删"## 我们如何发现这个流程"叙事

### 1.4 概念性解释

**结构特征**：

- "## 什么是 X"
- "## 为什么 X 存在"
- "## X 与 Y 的关系"

**改写目标**：参考文档加 "## 概念" 节，结构按 why / what / when 分。

**改写规则**：

- 概念名 + 一句话定义（≤ 30 字）
- 删除长篇的"我们理解 X 是..."——改为短陈述
- 引用（脚注 / "另见"）代替内联大段对比

### 1.5 已实施后的 gril 阶段 PLAN（最易被误切成多份）

**结构特征**：

- 源文件 100+ KB / 1500+ 行，含 `§0 决策记录` + `§接手必读` + `§N 实施步骤 P0-P8` + `§验收清单`
- 配套 README 是"实施过程"叙事而非"功能说明"
- 实施**已落地**，代码在 `config.org` / 蓝图 / 脚本里跑通

**这本质是 `SKILL.md §4.2 contract` 的特例**，不是 plan-to-doc 的标准 case（源不是"待实施的 plan"而是"已实施完的过程史"）。不在本节重复规则，速查：

1. 单文件 10-20 KB，匹配同仓库其他 docs
2. §0 决策表 + §接手必读 保留（**不重编号**）
3. §N 实施步骤 → §1 用法 + §3 已知陷阱（最致命 3-4 条）
4. §验收清单 → 删（已落地的历史在 git 里）
5. 代码块 ≤ 30 行，超出走"见 `source/config.org` `* <Topic>` 章节"

**反模式（高频踩坑）**：

- ❌ **自作主张切成多份 `docs/<topic>/`** —— 用户原话："我只需要一个用来说明相关功能的文档，参考其他文档的风格即可，不需要写非常多非常细，细的放到代码里面 (config.org) 解释会更好"。分多文件是 _response_，不是 _goal_
- ❌ 保留 "P0-P8 阶段" 叙事 —— 那是过程不是产品
- ❌ 复述 gril session 假设的"实施细节"（如 "PAM allow-empty-passwords #t"） —— 文档必须匹配现实代码，不是 gril 当时的猜测

**实证**（2026-07-07）：`docs/iso-build.md` 110 KB / 2155 行 → 12.8 KB / 268 行（88% 缩减），单 rewrite pass。读者按 "见 `source/config.org` `* Live ISO` 章节" 查代码细节，单一真相源保留。

## 2. 通用改写规则（所有类型）

### 2.1 Strip transient scaffolding

- 删时间戳（保留 "as of <date>" 或 "修订记录" 标记）
- 删调查过程（保留 "verified by X" 一行）
- **删 agent 推理**——读者看不到也不需要知道

### 2.2 Rewrite into target voice

| 源 PLAN 风格                         | 目标参考文档风格             |
| ------------------------------------ | ---------------------------- |
| 第二人称（"你"）                     | 第三人称（"用户"）           |
| 犹豫语气（"可能 / 也许 / 我们尝试"） | 陈述语气（"X 是 Y"）         |
| 内部对话（"我问了 A 才知道"）        | 公开结论（"X = Y，见 Z"）    |
| 时间序（"先 X 后 Y 最后 Z"）         | 主题序（"X 是；Y 是；Z 是"） |

### 2.3 Provenance footer

文末加：

```markdown
> 本节由 `<source-plan>.md` 在 <date> 提炼；原文归档在 <link>。
```

未来读者需要知道来源 PLAN 在哪儿、什么时候提炼的、有没有更权威版本。

### 2.4 Cross-link the source

在文档原章节加一个 "另见" 链接指回 PLAN。PLAN 留着能找到但不被读者直接看到。

## 3. 反模式

- ❌ 把 PLAN 原样搬进 docs/，加一行"这是 PLAN 转过来的"——读者读不到
- ❌ 只摘"好的"部分，"差的" 部分留在 PLAN——下次不知所以然
- ❌ 在参考文档里写"我们之前讨论过"——读者没上下文
- ❌ 改完不交叉链接——PLAN 变成孤儿
